# Repository Guidelines

## Project Structure & Module Organization
- `src/pages/`: route entry points (`.astro` files map to URLs). Example: `src/pages/about.astro`.
- `src/components/`: reusable UI components (PascalCase files like `TerminalHeader.astro`).
- `src/layouts/`: shared layouts such as `BaseLayout.astro`.
- `src/content/`: content collections and Markdown entries (kebab-case like `ai-project-will-fail.md`).
- `src/styles/`: global styling (see `src/styles/global.css`).
- `public/`: static assets served as-is (images, icons).
- `dist/`: production build output (generated by `npm run build`).

## Build, Test, and Development Commands
- `npm install`: install dependencies.
- `npm run dev`: start the Astro dev server at `http://localhost:4321`.
- `npm run build`: generate the production site into `dist/`.
- `npm run preview`: serve the `dist/` build locally for verification.
- `npm run astro`: access Astro CLI commands (e.g., `npm run astro -- check`).

## Coding Style & Naming Conventions
- Follow existing formatting in `src/` (no repo-level formatter is configured).
- Components: PascalCase filenames (e.g., `ProjectCard.astro`).
- Content entries: kebab-case filenames under `src/content/`.
- Prefer small, focused components and keep page files thin.

## Testing Guidelines
- No automated test suite is configured in this repo.
- If you add tests, document the framework and add a script in `package.json`.

## Commit & Pull Request Guidelines
- Commit history uses short, sentence-style summaries (e.g., "Initial Commit").
- Keep commits scoped and descriptive; avoid bundling unrelated changes.
- PRs should include a clear description of changes and screenshots for UI updates.

## Configuration Notes
- Site and build settings live in `astro.config.mjs` and `tsconfig.json`.
- When adding new content types, update `src/content/config.ts` accordingly.
